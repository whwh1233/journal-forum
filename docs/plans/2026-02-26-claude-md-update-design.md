# CLAUDE.md 更新设计文档

## 设计目标

更新 CLAUDE.md 使其能够在不同会话之间建立上下文连接，让每次新对话能够：
1. 快速了解项目功能全貌
2. 准确定位需要修改的模块和文件
3. 理解最近的改动历史和项目演进

## 设计原则

1. **最小改动原则** - 保持现有文档结构，仅增加必要的新区域
2. **快速定位优先** - 新增内容优先放置在文档前部
3. **维护成本低** - 每次更新只需修改 2 个区域（当前状态 + 最近改动）
4. **信息分层** - 避免重复，不同区域承担不同职责

### 新增内容 2：最近重要改动

**位置**: 文档末尾（在"TODO"列表之后）

**内容结构**:
```markdown
---

## 最近重要改动
> 记录最近 2-3 次会话的关键改动，便于新对话快速了解项目演进

### 2026-02-26 (commit: 670ae26)
图标系统重构 + UI 优化 + 组件测试覆盖（SVG→Lucide, SearchAndFilter 改进, ThemePicker/UserDropdown/Modal/BackButton/Breadcrumb 测试）

### 2026-02-XX (commit: xxxxx)
[上一次会话的改动摘要]

### 2026-02-XX (commit: xxxxx)
[再上一次会话的改动摘要]

---
**维护说明**: 每次会话结束时更新此区域，保留最近 3 次会话记录。
```

**设计要点**:
1. **日期 + commit hash** - 提供深入查看的入口（`git show <hash>`）
2. **一行式总结** - 用括号补充关键细节，保持 1-2 行内
3. **倒序排列** - 最新的在上面，方便快速扫描
4. **维护说明** - 提醒 Claude 每次更新的规则
5. **3 次会话限制** - 避免文档膨胀

**摘要格式**:
- 主要改动类型 + 技术决策 + 具体模块 + 附带成果
- 示例：`图标系统重构 + UI 优化 + 组件测试覆盖（SVG→Lucide, SearchAndFilter 改进, 5 个组件测试）`

## 信息架构协调

三个关键区域的职责分工：

| 区域 | 职责 | 回答的问题 |
|------|------|-----------|
| 功能模块导航 | 项目全貌索引 | "这个项目有什么功能？代码在哪里？" |
| 当前状态与待办 | 工作状态指示 | "现在处于什么状态？下一步做什么？" |
| 最近重要改动 | 历史演进记录 | "刚才做了什么？为什么这么做？" |

三者互补，新会话可以快速建立完整上下文：
1. 先看"功能模块导航"了解项目结构
2. 再看"当前状态与待办"了解工作重点
3. 最后看"最近重要改动"了解演进历史

## 维护流程

每次会话结束时，更新以下内容：

1. **更新"当前状态"** - 修改相关分类的状态描述
2. **更新"优先待办"** - 根据完成情况调整待办列表
3. **追加"最近重要改动"** - 添加本次会话的 1-2 行总结（包含日期和 commit hash）
4. **清理旧改动** - 保持"最近重要改动"只有 3 条记录

## 成功标准

新会话能够通过阅读 CLAUDE.md：
1. ✅ 在 30 秒内了解项目有哪些功能模块
2. ✅ 快速定位到需要修改的文件路径
3. ✅ 理解当前工作状态和优先事项
4. ✅ 了解最近 2-3 次会话的改动内容
5. ✅ 通过 commit hash 深入查看详细改动

## 实施要点

1. **保持简洁** - 功能模块导航每个模块不超过 8 行
2. **使用图标** - emoji 提升可读性和扫描速度
3. **提供路径** - 所有模块必须列出关键文件路径
4. **状态清晰** - 使用明确的状态标识（✅/🚧/❌）
5. **可追溯** - 最近改动必须包含 commit hash
